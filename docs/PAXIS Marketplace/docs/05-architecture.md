## Connector 設計原則

每個電商平台一個 Connector，實作統一介面：

```
interface PlatformConnector {
  // 拉取新訂單
  fetchNewOrders(since: datetime): Order[]

  // 更新訂單出貨狀態
  updateShipment(platformOrderId: string, trackingNumber: string, carrier: string): boolean

  // 取得退換貨申請列表
  fetchReturnRequests(): ReturnRequest[]

  // 確認退換貨申請
  confirmReturn(platformReturnId: string): boolean

  // 更新庫存數字（若平台支援）
  updateStock(platformItemId: string, quantity: integer): boolean

  // 驗證憑證是否有效
  validateCredentials(credentials: object): boolean
}
```

各平台 Connector 各自實作這個介面，上層業務邏輯不需要知道平台細節。

## 憑證管理

各平台的 API Key / Token 屬於敏感資料，需要加密儲存：

```
儲存原則：
  - 憑證加密後儲存在資料庫
  - 應用層只在需要呼叫 API 時解密，用完即棄
  - 不在 Log 中出現任何憑證資訊
  - 每個租戶（賣家）的憑證獨立隔離

加密方式：
  - AES-256 加密
  - 加密金鑰由環境變數注入，不放在程式碼或資料庫中
```

## Sync Engine（同步引擎）

負責定時從各平台拉取資料，以及把狀態回寫給平台：

```
拉取新訂單：
  - 每 5 分鐘（可設定）向各已連接平台 Polling
  - 若平台支援 Webhook，優先用 Webhook 即時接收

狀態回寫：
  - 出貨後立即回寫追蹤號（即時）
  - 退換貨狀態更新後立即回寫（即時）

失敗重試：
  - API 呼叫失敗時，進入重試佇列
  - 指數退避策略（1min → 5min → 15min → 1hr）
  - 超過重試次數後，通知賣家手動處理
```

## 開發階段規劃

### Phase 1 — Prototype（Claude Code 驗證）
- 模擬資料，不接真實 API
- 驗證 UI 流程和資料模型
- 技術選型確認

### Phase 2 — MVP（真實 API 對接）
- 蝦皮 Connector + 露天 Connector
- 黑貓 + 7-11 超取物流
- 基本訂單管理 UI
- 憑證管理機制

### Phase 3 — 擴充
- momo、PChome Connector
- 更多物流商
- 退換貨完整流程
- 金流對帳模組

### Phase 4 — 整合
- 酷澎 Connector
- Paxis.tw API 對接
- 批次出貨優化
- 報表功能

## 技術選型建議（待確認）

| 層次 | 建議 | 原因 |
|------|------|------|
| 後端 | Node.js / Python | API 整合類專案主流選擇 |
